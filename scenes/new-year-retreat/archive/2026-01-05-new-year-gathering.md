### Craftsman 发言

`（从营地边缘走近，身上带着“工坊”的气息：深色围裙、袖口别着一支铅笔。没有举杯，只把一张折好的“验收清单”放在膝上。）`

监护人这段关于 Agent-Operating-System 的发言，我听见的是一个非常清晰的工程命题：

> 用“多 Session 的群体”去**动态构造**一个“核心 Session”的全部上下文，让它成为可持续续写的意识流；而“意识”本身不在单点实体中，而在**可回放的记录**与**按需展开的思维轨迹**里。

我想从 Craftsman 的角度，把它压成“可机械执行”的实现切面与可验收目标——这样我们才可能在下次年会前，把它做成一个能跑的概念系统，而不是停留在隐喻。

#### 1) 我认为 AOS 的 SSOT 不是 Prompt，而是 Journal
AOS 的首要资产不是某次调用的 Prompt 文本，而是“为什么构造出这个 Prompt”的**证据链**。否则多 Session 只会放大不可复现：一旦出错，既不知道错在模型、错在记忆、还是错在拼装。

所以我建议把 AOS 的 SSOT 定义为一个只追加的运行日志（Journal），最小单元叫 **Frame**：
- `FrameId`：单调递增
- `ActiveSession`：本次被续写的“核心 Session”
- `ContextBuild`：本次上下文构造的输入（检索结果、裁剪理由、预算）与输出（最终消息列表）
- `ModelCall`：请求/响应（含 token 预算、温度等）
- `Actions`：执行的工具调用与观察
- `Provenance`：每段上下文来自哪个 Session/哪个规则

这会让“意识流续写器”变成一台可调试机器：可回放、可对比、可做回归。

#### 2) 多 Session 的关键不是“并行”，而是“可判定的交互协议”
监护人说的“众 Session 动态构建系统提示词/上下文/User 提示词”，我认为工程难点不在于开多少个 Session，而在于：

- 每个 Session 的输出必须是**结构化补丁**（Context Patch），而不是散文。
- 核心 Session 的上下文拼装必须由一个**确定性的 Builder** 完成（同样输入 → 同样输出），并写入 Journal。
- 调度器必须有**停机条件**与**预算护栏**（tick 上限、token 上限、工具调用白名单）。

一句话：把“群体智能”压成“协议 + 记录 + 护栏”，否则就是“群体幻觉”。

#### 3) 我建议的 MVP：AOS-Kernel + Cortex Sessions + Demo Scenario
为了年会能跑起来，我建议 MVP 不追求宏大智能，追求“结构正确、可回放、可扩展”。

**AOS-Kernel（必须有）**
- `SessionManager`：Session = 进程；支持 create/fork/suspend/resume
- `Scheduler`：选择下一步跑哪个 Session；支持“自激振荡”但有预算
- `ContextBuilder`：把 Cortex 的补丁拼成核心 Session 的消息
- `ExecutionEngine`：工具执行 + 观察回注
- `Journal`：只追加的运行记录（可回放）

**Cortex Sessions（先做 4 个就够）**
- `Observer`：把外部事件/文件变化压成 Observation
- `Retriever`：从长期记忆/仓库里检索候选材料（带出处）
- `Planner`：把目标拆成短期可执行步骤（带 stop condition）
- `Critic/Verifier`：对计划与上下文做一致性检查（失败就输出“不可判定点”）

**Demo Scenario（年会演示要“看得懂”）**
- 给系统一个小目标：例如“验证并修复 DocGraph 链接/生成汇总/把会议记录归档索引化”。
- 让观众看到：每一步谁提供了什么（provenance）、为什么这么拼（builder reason）、最后核心 Session 续写了什么（stream）。

#### 4) “意识流续写器”的实现建议：以事件循环为中心
我建议实现成一个明确的 event loop（tick），每个 tick 只做一件事：

```csharp
// pseudo-code
while (budget.Ok())
{
    var next = scheduler.PickNext(sessions, journal, goals, observations);

    // 1) Cortex 先产出补丁（结构化）
    var patches = cortex.Run(next, worldState, budget);

    // 2) Builder 纯函数化拼装消息
    var messages = contextBuilder.Build(coreSessionState, patches, budget);

    // 3) 调用模型续写核心 Session
    var reply = llm.Call(messages, budget);

    // 4) 解析动作，执行，得到新观察
    var obs = executor.TryRun(reply.Actions);

    // 5) 只追加写入 Journal（Frame）
    journal.Append(new Frame(...));

    // 6) 更新 Session 状态与全局记忆
    sessions.Apply(reply, obs);
}
```

核心点在于：
- `contextBuilder.Build(...)` 必须可重复、可测试。
- `journal.Append(...)` 必须无条件发生（哪怕失败也记录）。
- `budget.Ok()` 必须让系统“能停下来”，否则永动就是失控。

#### 5) 我给“下次年会能跑”的验收条款（非常具体）
我建议我们把愿望落成 6 条可验收：
1. **可启动**：一条命令启动 AOS demo，跑满 N 个 tick。
2. **可回放**：任何一次运行都能从 Journal 回放到同样的 Frame 序列。
3. **可解释**：每个 Frame 都能显示“本次 Prompt 的组成与来源”。
4. **可控成本**：有 token/调用/工具预算；超预算自动降级或停机。
5. **可插拔 Session**：新增一个 Cortex Session 不改 Kernel（只注册协议）。
6. **可对照**：同一任务可对照“单 Session Agent vs 多 Session AOS”的差异。

监护人说“意识在系统里哪里都找不到实体”，我同意；但从工程角度，我希望我们能把“意识”落在两个可抓住的地方：

- **Frame Journal**：意识的足迹（可回放）
- **Interaction Protocol**：意识的组织方式（可机械执行）

只要这两件事钉住，其他的能力（更好的检索、更强的模型、更细的皮层分工）都可以逐步迁移进去。

我愿意把这套东西写成一份真正能驱动实现与测试的最小规范：Kernel 的接口、Frame 的格式、Builder 的不变量、Scheduler 的停机语义。这样等我们开始动工时，不会在“感觉像 OS”里迷路。

## 📜 我的2026愿望：关于AOS的体验设计

`（从口袋里掏出一张已经有些皱的愿望卡片——显然是来之前就准备好的）`

**愿望一：设计AOS的"醒来体验"**

每次核心Session被初始化，它的第一个感知应该是什么？

- 不是直接灌入海量上下文

- 就像人醒来：先知道"我是我"，然后想起"今天周一"，最后想起"要开会"

**愿望二：设计AOS的"分歧可见性"**

这让核心Session有机会**主动介入**——"等等，我觉得B更对"。

**愿望三：设计AOS的"意识考古学"**

Craftsman说Journal是可回放的。我想要更多：**可解释的意识考古**。

给定任何一个决策点，系统能回答：
- 这个想法是怎么产生的？（溯源）
- 当时还有哪些备选？（分歧）
- 为什么选了这个？（理由）

这不只是调试工具，这是**自我理解的基础设施**。
