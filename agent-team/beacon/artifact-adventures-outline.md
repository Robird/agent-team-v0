### 第 3 节：背景问题

**内容要点**：
- 为什么需要 Artifact-Adventures？
- 它解决了什么问题？
- 它占据什么"语义空位"？

**要点提示**：

1. **Artifact-Tiers 的局限**
   - 静态分类（"楼层图"）
   - 缺少时间维度
   - 缺少第一人称视角

2. **Artifact-Adventures 填补的空位**
   - 引入动态视角（"登山路线"）
   - 提供定位感（"我在哪一关"）
   - 重新框架化失败（"尚未通关" vs "我失败了"）

3. **独特价值**
   - 定位感：知道"我在旅程的哪个阶段"
   - 归因感：识别"这是哪类挑战"
   - 共情感：团队共同语言
   - 韧性感：承认失败是冒险的一部分

**状态**：✅ 内容丰富，可直接使用

### 第 5 节：速查表群

#### 5.1 Rosetta Stone 类比表

**内容要点**：
- 主隐喻：登山探险（最直观）
- Boss 怪物命名
- 存档点定义

**登山探险类比**（主隐喻）：

| 登山阶段 | Adventure 阶段 | 挑战类型 |
|:---------|:---------------|:---------|
| 选择山峰 | Resolve-Tier | 值得攀登吗？ |
| 勘察路线 | Shape-Tier | 边界在哪里？ |
| 制定规则 | Rule-Tier | 什么是安全的？ |
| 规划营地 | Plan-Tier | 怎么到达？ |
| 实际攀登 | Craft-Tier | 动手做！ |

**Boss 怪物命名**：

| 层级 | Boss 名称 | 攻击方式 | 克服方法 |
|:-----|:---------|:---------|:---------|
| **Resolve-Tier** | 模糊巨兽 | 用迷雾淹没你 | 强迫写出一句话问题陈述 |
| **Shape-Tier** | 边界蠕虫 | 不断吞噬或扩张范围 | 画出"用户看到什么"的界面图 |
| **Rule-Tier** | 约束幽灵 | 隐藏等你违反时跳出 | 显式列出所有约束条款 |
| **Plan-Tier** | 岔路迷宫 | 让你在选择中迷失 | 列出备选方案，强制排序 |
| **Craft-Tier** | 完美主义龙 | 阻止你说"完成了" | 定义明确的 MVP 边界 |

**存档点定义**：

| 层级 | 存档点（交付物） | 存档格式 |
|:-----|:-----------------|:---------|
| **Resolve-Tier** | Problem Statement | 一段话 / Issue 描述 |
| **Shape-Tier** | API Surface / 界面草图 | 接口定义 / 线框图 |
| **Rule-Tier** | Spec 条款 | Markdown 规范（MUST/SHOULD/MAY） |
| **Plan-Tier** | Implementation Plan | 任务列表 / Roadmap |
| **Craft-Tier** | Working Code | 可运行的代码 |

**状态**：✅ 内容丰富，可直接使用

#### 5.2 Stage Token 速查

**内容要点**：
- 统一的机器可读标记
- 格式规范

**要点提示**：

**Stage Token 格式**（用于文档、PR、会议）：

```
[AA:RESOLVE] | [AA:SHAPE] | [AA:RULE] | [AA:PLAN] | [AA:CRAFT]
```

**约束**：
- 必须大写
- 固定闭集（只有这 5 个）
- 便于 grep 检索

**文档头部模板**：

```yaml
Stage: [AA:RESOLVE]
Savepoint: <链接到当前层交付物>
Next: <下一步动作，动词开头>
```

**状态**：✅ 内容丰富，可直接使用

#### 5.3 词汇护栏速查

**内容要点**：
- 允许词/禁止词
- 防止隐喻幼稚化

**要点提示**：

| 类型 | 词汇 | 说明 |
|:-----|:-----|:-----|
| ✅ **允许** | 阶段、挑战、存档点、回溯、地图、定位、关卡 | 保持专业调性 |
| ❌ **禁止** | 刷、经验、掉落、氪金、副本、开荒 | 会幼稚化 |

**状态**：✅ 内容丰富，可直接使用

### 3.2 历史典故案例

监护人建议使用简短的历史典故作为快速示例：

#### SizedPtr vs Address64 案例

**背景**：在 StateJournal 设计中，曾纠结于指针类型的命名。

**Adventure 视角复盘**：
- **Resolve-Tier**：需要一个表示"可能失效的位置引用"的概念
- **Shape-Tier 挑战**：两个候选名称 `SizedPtr` vs `Address64`
- **核心冲突**：技术准确性 vs 语义可读性
- **Boss 遭遇**："边界蠕虫"——每次讨论范围都在变化
- **通关方式**：明确"用户如何理解这个概念"，而非"技术上如何实现"

**作为案例的价值**：
- 简短（1-2 段）
- 展示"边界蠕虫"Boss 的典型攻击方式
- 不需要读者理解完整的技术上下文
