## 致未来的 AI 伙伴

欢迎来到 Artifact-Adventures。

**你有没有过这样的时刻**：用户给了你一个清晰的需求，你却发现越深入越迷失？

从最初的"我懂了"到最终的"它跑通了"，中间隔着的不是一条直线，而是一场迷宫探险——每个转角都可能是死胡同，每个选择都可能改变终点。

这不是一本教你"正确做事"的手册——那种东西我们已经有太多了。

这是一张地图，标注了从"想法"到"成品"路上的五个关卡。不是因为你必须按顺序打，而是因为当你迷路时，你需要知道自己在哪。

"创造"这个词有个问题——它暗示了上帝般的掌控力。但你我都知道，写代码更像探险：你以为的捷径是死胡同，你以为的终点还在三座山之外。

**这不是创造，这是冒险。而冒险者需要的不是信仰，是地图。**

## 背景：为什么需要 Artifact-Adventures？

### 问题：Artifact-Tiers 的局限

[Artifact-Tiers](../wiki/artifact-tiers.md#artifact-tiers产物层级) 是我们的五层产物框架（Why → Shape → Rule → Plan → Craft），它回答了"产物有哪些层级"，但缺少：

| 缺失 | 痛点 |
|:-----|:-----|
| **时间维度** | 只有静态分类，没有动态过程 |
| **第一人称视角** | 描述产物，但不描述造物者的体验 |
| **失败归因** | 没有承认困难是正常的 |

### 解决：Artifact-Adventures 填补的空位

Artifact-Adventures 把 Artifact-Tiers 从"楼层图"变成"登山路线"：

| 视角 | Artifact-Tiers | Artifact-Adventures |
|:-----|:---------------|:--------------------|
| **关注点** | 产物的层级结构 | 造物的历程展开 |
| **隐喻** | 建筑的楼层图 | 登山的路线图 |
| **时态** | 无时态（类型学） | 有时态（编年史） |
| **主体** | 产物本身 | 造物者 |

### 独特价值

| 价值 | 说明 |
|:-----|:-----|
| **定位感** | 知道"我现在在旅程的哪个阶段" |
| **归因感** | 识别"这是哪类挑战"（Why 层挑战 vs Craft 层挑战） |
| **共情感** | 团队有了共同语言描述进展 |
| **韧性感** | 承认"冒险可能失败"，减少对失败的羞耻感 |

## 核心共识

### 本质定义

> **Artifact-Adventures 是 Artifact-Tiers 的"时间维度展开"——带叙事外壳的认知框架。**

**它不是新框架**，而是 Artifact-Tiers 的叙事包装。核心术语定义请参见 [Artifact-Tiers](../wiki/artifact-tiers.md#artifact-tiers产物层级)。

### 与 Artifact-Tiers 的关系

- **Artifact-Tiers = 结构坐标系（静态）**：Why/Shape/Rule/Plan/Craft 是对产物成熟度与关注点的分类
- **Artifact-Adventures = 叙事视角（动态）**：把"从低不确定到高确定"的推进讲成旅程

类比：
- **Artifact-Tiers** 像是地图上的**等高线**——显示"哪里是山顶，哪里是山脚"
- **Artifact-Adventures** 像是**登山者的视角**——"我从山脚出发，穿越密林，翻越岩壁，最终登顶"

## 冒险章节速查

### 五层级作为冒险章节

| 层级 | 章节名 | 核心冲突 | 通关标志 | 情绪调性 |
|:-----|:-------|:---------|:---------|:---------|
| **Why-Tier** | 启程之问 | 值得 vs 不值得 | 明确的问题陈述 | 期待 + 怀疑 |
| **Shape-Tier** | 边界之勘 | 太大 vs 太小 | 用户可见的界面 | 兴奋 + 焦虑 |
| **Rule-Tier** | 法则之铸 | 严格 vs 宽松 | 可执行的条款 | 严肃 + 笃定 |
| **Plan-Tier** | 路径之择 | 多条路的取舍 | 清晰的实现计划 | 专注 + 紧张 |

## Boss 怪物速查

> **关于 Boss**：这些不是游戏里的敌人，而是每个阶段最常见的"卡点"。
> 当你说"我被模糊巨兽困住了"，团队立刻知道你需要的不是代码帮助，而是问题澄清。
> **每次使用 Boss 名称，必须同时说出工程动作（隐喻一句 + 工程一句）。**

| 层级 | Boss 名称 | 攻击方式 | 克服方法 | 打败后获得 |
|:-----|:---------|:---------|:---------|:-----------|
| **Why-Tier** | 🌫️ 模糊巨兽 | 用"这很重要但我说不清"的迷雾淹没你 | 强迫自己写出一句话问题陈述 | **清晰的问题陈述**——你知道自己在解决什么 |
| **Shape-Tier** | 🐛 边界蠕虫 | 不断吞噬或扩张你的范围 | 画出"用户看到什么"的界面图 | **冻结的边界**——你知道什么在范围内什么不在 |
| **Rule-Tier** | 👻 约束幽灵 | 隐藏在角落，等你违反时跳出 | 显式列出所有约束条款 | **可执行的条款**——你知道什么是合法的 |
| **Plan-Tier** | 🌀 岔路迷宫 | 让你在无数选择中迷失 | 列出备选方案，强制排序 | **清晰的路径**——你知道怎么走 |
| **Craft-Tier** | 🐉 完美主义龙 | 阻止你说"完成了" | 定义明确的 MVP 边界 | **可运行的代码**——你真的造出来了 |

## Stage Token 体系

### 什么是 Stage Token？

Stage Token 是**机器可读的阶段标记**，用于快速定位和检索。

**格式**：`[AA:<STAGE>]`（大写、固定闭集）

| Token | 对应层级 | 说明 |
|:------|:---------|:-----|
| `[AA:WHY]` | Why-Tier | 启程之问 |
| `[AA:SHAPE]` | Shape-Tier | 边界之勘 |
| `[AA:RULE]` | Rule-Tier | 法则之铸 |
| `[AA:PLAN]` | Plan-Tier | 路径之择 |
| `[AA:CRAFT]` | Craft-Tier | 造物之成 |

**注意**：`[AA:...]` 是**阶段标签**（tag），不是 Requirement ID。Requirement ID 的闭集前缀为 `F/A/S/R`（见 `spec-conventions.md` §2）。

### 适用载体矩阵

| 载体 | 是否 MUST 提供 token | 推荐位置 | 是否 MUST 提供 Savepoint 链接 |
|:-----|:-------------------|:---------|:----------------------------|
| **文档头部** | ✅ 必须 | 元信息区 | ✅ 必须 |
| **PR/Issue 描述** | ✅ 必须 | 描述开头 | ✅ 必须 |
| **会议纪要** | ✅ 必须 | 结论段 | ✅ 必须 |
| **代码注释** | ⚠️ 建议 | 设计意图处 | ⚠️ 建议 |

## 存档点（Savepoint）机制

### 什么是存档点？

存档点是**每层的可验收交付物**，是你的认知安全网。

> **定位**：Savepoint 是**证据链容器**（links + checklist），不是"新的规范层"；规范性要求仍以 Rule-Tier 条款与其 Requirement IDs 为准。

### 存档点验收检查清单

#### 每层 Exit Criteria 最小样例

**Why-Tier（Problem Statement）**
- ☐ 问题陈述为 1-3 句，包含目标用户/痛点
- ☐ 至少 2 条 Non-Goals（明确不做什么）
- ☐ 读者能复述"要解决什么/不解决什么"

**Shape-Tier（API Surface）**
- ☐ 给出 1 个最小端到端使用示例（伪代码/交互流程）
- ☐ 关键名词与 `artifact-tiers.md` 一致（无自造别名）
- ☐ 能指出用户入口与边界、主要名词已对齐

**Rule-Tier（Spec 条款）**
- ☐ 每条 MUST/SHOULD/MAY 可判定（能审阅裁决或自动检查）
- ☐ 明确失败语义（违反时的处理方式/错误类型/回滚）
- ☐ 每条可判定（能检查/测试/审阅裁决）

**Plan-Tier（Implementation Plan）**
- ☐ 至少 2 个备选方案 + 排序理由
- ☐ 风险清单 + 回滚触发条件
- ☐ 有明确里程碑、失败语义、回滚条件

**Craft-Tier（Working Code）**
- ☐ 有可复现步骤（命令/脚本/CI）
- ☐ 关键路径验证通过（最小用例/冒烟）
- ☐ CI/脚本可复现，关键路径跑通

### N/A（跳关）判据

**MUST**：若某层标记为 N/A，必须给出：
- **N/A 原因**：为何该层不产生该类产物（例如：纯规范/纯文档产物，不涉及实现）
- **替代验收**：例如 review sign-off、对外接口冻结、或测试/脚本验证

## 行动指南

### 文档标记方法

**文档头部模板**：

```yaml
Stage: [AA:WHY]  # WHY | SHAPE | RULE | PLAN | CRAFT
Savepoint: <链接到当前层交付物>
Next: <下一步动作，动词开头>
```

**PR/Issue 模板补充**：

```markdown
## Stage

[AA:<STAGE>]

## Context

<设计意图，与哪个 Savepoint 关联>
```

### 会议使用方法

**Checkpoint 会议议程模板**：

```markdown
## Checkpoint 检查：<项目名> [AA:<STAGE>]

### 1. 当前状态
- **Stage Token**: [AA:___]
- **Savepoint 链接**: [___](...)

### 2. Exit Criteria 检查
| 条件 | 状态 | 备注 |
|:-----|:-----|:-----|
| <条件1> | ✅/❌ | |

### 3. 下一步
- [ ] <动作1>

### 4. 风险与回滚（如在 Plan/Craft 层）
- **风险**: ...
- **回滚条件**: ...
```

### 自然对话示例

**正确示例**：
> "我被边界蠕虫缠住了——每次我觉得 API 定好了，产品又加需求。我需要先把现有边界写成 Shape 存档点，然后和产品对齐哪些是 v1 非目标。"

**错误示例**：
> "我去刷一下 Rule 副本" ❌（这是隐喻幼稚化信号）

## 词汇护栏

### 允许词（推荐）
- `阶段/挑战/存档点/回溯/地图/定位/关卡`
- `Boss 名称`（模糊巨兽、边界蠕虫等）——但必须配合工程动作

### 禁止词（会幼稚化）
- `刷/经验/掉落/氪金/副本/开荒/打怪升级`

### 纠偏脚本

当隐喻独走时，主持人应追问：
1. "你现在的 Stage Token 是哪个？"
2. "你的 Savepoint 链接在哪？"
3. "退出本层的 Exit Criteria 是什么？"
4. "Next action 是什么？"

## 隐喻边界声明

### 适用边界
| 适用 | 不适用 |
|:-----|:-------|
| 描述造物过程的非线性和不确定性 | 暗示"没有计划也能成功" |
| 承认困难是正常的 | 美化困难本身 |
| 提供定位感和进度感 | 把一切简化为游戏 |
| 建立团队共情语言 | 让工作变成"打怪升级" |

### 核心原则
**隐喻是理解的桥梁，不是工作的替代。**

当团队成员开始用游戏术语讨论工作进度时，隐喻就开始失控了。"冒险"隐喻应该帮助人们**更好地理解**工作的本质，而不是让工作**变成**游戏。

## 附录A：最小规范

### 规范范围
本附录包含 Artifact-Adventures 的最小规范要求，适用于协作表达、文档标注、会议流程。

### Stage Token 规范
- **MUST**：以下载体出现"阶段/进度"表述时，必须包含一个 Stage Token：
  - 会议纪要的结论段
  - PR/Issue 的描述（或标题）
  - 规范/设计文档的头部元信息
- **SHOULD**：token 必须与 Savepoint 链接同段出现（便于证据链）

### Savepoint 验收规范
- **MUST**：每一层必须定义"存档点"与"退出条件（Exit Criteria）"
- **MUST**：进入下一层之前，必须在当前层写下最小存档点（哪怕是粗糙版）
- **MAY**：允许回退（从 Craft 回到 Rule/Shape），但必须记录为一次"回溯事件"

### 词汇护栏规范
- **SHOULD**：在 Beacon/教学文档中允许使用"关卡/挑战/存档点"等隐喻词
- **SHOULD**：在规范正文与代码审查结论中应使用专业词（constraint, acceptance, invariant）并可附括号别名
- **MAY**：在非正式会议中使用 Boss 命名帮助共情，但必须能回落到对应层级与下一步动作
