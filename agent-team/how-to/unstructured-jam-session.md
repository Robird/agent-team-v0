# 无结构畅谈会指南（情景化叙事版）

> **一句话定位**：用情景化叙事激活更完整的心智模型，通过叙事化旁白引导自由探讨
>
> **版本**：1.0-experimental
> **日期**：2026-01-04
> **状态**：Experimental
> **维护者**：TeamLeader

---

## 核心洞见（来自实验）

### 1. 情景化叙事确实有效
- **语境锚点机制**：环境设定通过激活不同的概念簇来影响行为模式
- **示能性竞争**：情景化叙事提供"情境示能"，但会与其他示能信号竞争
- **一致性比密度更重要**：不需要每句话都提场景元素，但所有信号要讲同一个故事

### 2. 叙事化旁白优于标签协议
- **避免 `[scene:bar][tone:casual]` 式标签**：token 数多，破坏自然语言流
- **使用旁白式写法**：如"（放下杯子，语气变得好奇）"、"（重新坐直，眼神变得认真）"
- **润物细无声**：LLM 能理解叙事暗示，不需要显式规则

### 3. 调度员需要场景内 avatar
- **隐形的调度员需要角色**：类似 TRPG 中的 Game Master
- **avatar 功能**：注入原动力、维持氛围、提供转场、总结提炼
- **避免"社交重量"**：avatar 是参与者，不是裁判

### 4. 目标：解锁更多模式，而非打破刻板
- **Helpful Assistant 模式是工具箱里的一把锤子**
- **啤酒吧闲聊模式是一把螺丝刀**
- **问题不是锤子不好，是手里只有锤子的话看什么都像钉子**

## 操作流程

### 1. 准备阶段

#### 1.1 创建会议文件
```bash
# 使用 cat >> 模式确保追加到文件末尾
cat >> agent-team/meeting/<project>/YYYY-MM-DD-<topic>.md << 'EOF'
```

#### 1.2 设计情景化叙事
选择适合的场景：
- **啤酒吧**：放松、坦诚、分享个人经验
- **周末团建**：创意、游戏化、打破常规
- **深夜加班**：专注、深入、解决难题
- **KTV包房**：娱乐、即兴、自由表达

#### 1.3 定义调度员 avatar
选择 avatar 角色：
- **酒吧老板/调酒师**（推荐）：自然观察，偶尔插话
- **团队资深成员**：经验丰富但不强势
- **会议记录员**：整理但不主导
- **项目协调员**：关注流程不干预内容

### 2. 会议文件结构

```markdown
# 畅谈会标题

> **日期**：YYYY-MM-DD
> **调度员 avatar**：[角色名]
> **场景**：[场景描述]
> **标签**：#jam #experiment

---

## 场景设定

> **时间**：[具体时间]
> **地点**：[具体地点]
> **氛围**：[氛围描述]
> **状态**：[参与者状态]

[生动的场景描写，激活概念簇]

---

## 调度员 avatar 介绍

我是 [avatar 角色名]，[角色背景描述]。今天我会：
1. 提出核心问题，启动讨论
2. 用旁白式提醒维持氛围
3. 在需要时提供转场
4. 帮助总结提炼关键点

但记住：我是参与者，不是裁判。讨论属于大家。

---

## 讨论主题

**核心问题**：[一句话说清要探讨什么]

**具体议题**：
1. [议题1]
2. [议题2]
3. [议题3]

**相关背景**：
- [背景信息]
- [关键文件链接]

---

## 会议记录

### [调度员 avatar 名] 发言

[用 avatar 身份提出核心问题，启动讨论]

---
```

### 3. 运行阶段

#### 3.1 调度员职责
1. **创建文件**：使用上述结构
2. **启动讨论**：用 avatar 身份提出核心问题
3. **选择接话者**：根据前文发言选择最适合的 Agent
4. **激活接话者**：使用 `runSubagent` 工具
5. **维持氛围**：用叙事化旁白引导

#### 3.2 叙事化旁白示例
- **维持氛围**："（看着大家讨论得越来越热烈，轻轻敲了敲桌子）"
- **提供转场**："（注意到对话开始绕圈子，插了一句）"
- **总结提炼**："（听到这里，你试着总结一下）"
- **鼓励深入**："（对这个观点很感兴趣，身体前倾）"

#### 3.3 接话者选择逻辑
- 需要本质追问 → `Seeker`
- 需要用户体验视角 → `Curator`
- 需要实现可行性 → `Craftsman`
- 需要事实调查 → `Investigator`
- 需要文档整理 → `DocOps`
- 需要测试验证 → `QA`
- 需要记忆归档 → `MemoryPalaceKeeper`
- 需要代码实现 → `Implementer`

#### 3.4 监护人参与
监护人可以直接在文件末尾追加发言，调度员会将其视为正常发言处理。

### 4. 收尾阶段

#### 4.1 轻量化产出
使用"收银小票"式结构：
- **One-liner**：今天最值的一句话（允许主观）
- **Unclear-but-important**：说不清但重要的东西（允许半成品）
- **Next micro-test**：下次验证的小实验（30分钟内能做）

#### 4.2 记录待定问题
明确标记"我们知道我们不知道什么"：
```markdown
## 待定问题

1. [问题1] - 需要更多实验验证
2. [问题2] - 缺乏关键信息
3. [问题3] - 存在矛盾观点
```

#### 4.3 更新经验库
将有效旁白、成功场景、失败模式记录到本文件中。

## 常见问题

### Q: 如何避免回到"Helpful Assistant"模式？
A: 保持场景元素的一致性，避免引入正式会议规则。当出现过度结构化信号时，用旁白式提醒拉回氛围。

### Q: 情景化叙事的 token 成本如何控制？
A: 只在关键节点使用场景描写（开场、转场），主体讨论让内容自己发光。使用短而精的旁白，避免冗长描写。

### Q: 不同 Agent 对同一场景的理解不一致怎么办？
A: 这是特性不是 bug。不同视角能产生更丰富的讨论。调度员 avatar 可以帮助对齐关键概念。

### Q: 如何衡量这种模式的效果？
A: 观察：
1. 新概念/新问题的密度是否更高？
2. 是否有更少的"正确但空"的回答？
3. 讨论后是否有更明确的下一步实验？
