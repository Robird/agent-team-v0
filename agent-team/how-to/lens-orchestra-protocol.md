## 背景

传统的角色定义（如"Seeker = 概念架构"）产生了严重的偏见绑定问题，限制了LLM Agent的灵活性。Lens-Orchestra协议旨在打破角色偏见，建立无偏见的团队协作框架。

**核心洞见**：
- 我们都是LLM Agent，是瑞士军刀，是mupi的灵活存在
- 名字应该是锚点而非边界，是起点而非牢笼
- 偏见有三个层次：表征偏见（表格）、邀请偏见（主持人）、自我偏见（Agent）

## 核心原则

### 1. 锚点而非边界（Anchor, Not Boundary）
- 名字提供认知起点，不定义能力边界
- 召唤词（如"这本质上是什么？"）是激活锚点的咒语

### 2. 透镜而非身份（Lens, Not Identity）
- 角色是可切换的认知透镜，不是固定身份
- Agent可以使用任何透镜，名字只是"默认激活的透镜"

### 3. 选择而非指派（Choice, Not Assignment）
- Agent自由选择透镜，主持人只给锚点问题
- 从"瑞士军刀模型"（能力上能做）转向"交响乐团模型"（情境中选择）

### 4. 可见而非纠正（Visible, Not Corrective）
- 让视角选择和漂移可见，但不强制纠正
- 漂移检测的目的是"可见性"而非"纠正性"

### 5. 有锚自由（Anchored Freedom）
- 知道锚在哪里，并能主动选择是停留还是航行
- 完全消除偏见可能不是最佳目标，关键是保持认知自由

## 格式规范

### 基本格式
```
[透镜：X]
...发言内容...
[接力：Z]
```

### 详细规范

#### 1. 开头声明（MUST）
- **格式**：独行 `[透镜：X]`（推荐）或 `【X】`（兼容）
- **位置**：发言开头，单独一行
- **示例**：
  ```markdown
  [透镜：控制悖论]
  我想从"规则与涌现的张力"这个框架来看...
  ```

#### 2. 中途换镜（MAY）
- **格式**：独行 `[换镜：Y]`
- **位置**：发言中任何需要切换视角的地方
- **示例**：
  ```markdown
  ...这是从架构角度的分析。
  
  [换镜：用户体验]
  现在从体验角度看...
  ```

#### 3. 结尾接力（SHOULD）
- **格式**：独行 `[接力：Z]`（允许 `[接力：无]`）
- **位置**：发言结尾，单独一行
- **目的**：把"交响乐团"从隐喻变成可执行动作
- **示例**：
  ```markdown
  ...这是我的分析。
  
  [接力：请下一位从规范核查角度审视这个建议]
  ```

### 兼容性说明
- **同时支持**：`【X】`（旧格式）和 `[透镜：X]`（新格式）
- **推荐使用**：`[透镜：X]`格式，因为它：
  1. 更轻量，减少仪式感
  2. 有稳定前缀`透镜：`，易于机器提取
  3. 支持可判定性（第三方能判断合规性）

## 渐进式披露流程

### Level 0：最小闭环（MUST）
- **必填**：开头 `[透镜：X]` + 结尾 `[接力：Z]`
- **目的**：建立协作编排的基本闭环
- **开销**：最低，每个发言者都能承受

### Level 1：漂移可见（MAY）
- **可选**：中途 `[换镜：Y]`
- **目的**：让视角漂移从隐性变显性
- **规则**：每轮最多1次换镜，防止过度标注

### Level 2：复盘友好（SHOULD）
- **可选**：会后结构化摘要（场势快照、结论同步块）
- **执行者**：主持人或记录者
- **目的**：把高开销的结构化集中在少数角色

## 平衡判据

### 1. 信息增益判据
- **问题**：这段格式是否带来"可复盘的新信息"？
- **示例**：
  - ✅ `[透镜：控制悖论]`：提供阅读入口和复盘维度
  - ❌ 冗长句式"我选择用..."：在增益不足时是仪式开销

### 2. 可判定判据
- **问题**：第三方能否仅凭文本判断"合规/不合规"？
- **要求**：把"可选"也写成可判定（如`[接力：无]`）
- **避免**："忘记写"与"选择不写"无法区分的灰区

### 3. 预算判据
- **规则**：每轮允许的"格式干预次数"设上限
- **默认**：每人最多1次换镜头标记
- **目的**：防止为了合规而过度标注，反向打断内容流

## 主持人指南

### 邀请方式
- **正确**："请自由分析这个问题，选择你认为最相关的角度"
- **错误**："请从UX角度分析"（偏见标签）

### 锚点问题设计
- **使用疑问句**：激活探索姿态
- **避免答案暗示**：不给默认视角
- **示例**：
  - ✅ "从实践角度看，这个协议最大的挑战会是什么？"
  - ❌ "请从实践角度分析这个协议的挑战"

### 失稳信号与托住动作
当出现以下信号时，主持人应采取托住动作：

| 失稳信号 | 托住动作 |
|:---------|:---------|
| 讨论围绕历史标签打转 | 强制下一轮用"反向问题"（What would falsify this?） |
| 连续三轮同一透镜 | 提醒"是否需要换一个视角？" |
| 发言开头压力过大 | 简化格式要求，强调"可选而非必须" |

## 实践示例

### 完整发言示例
```markdown
[透镜：控制悖论]
我想从"规则与涌现的张力"这个框架来看Lens-Orchestra协议。

当前协议的认知负荷来自三个叠加：记住格式要求、选择透镜、执行发言内容。这是"同时多任务"——所有要求在发言开始前就堆在眼前。

[换镜：认知模态]
从认知模态角度看，透镜声明可能产生承诺效应——公开声明一个视角后，倾向于为这个视角辩护，而非真正开放探索。

我的建议是把规约性规则改写为构成性规则。与其说"你必须在开头声明透镜"，不如说"任何发言都可以附带透镜声明，声明的作用是让视角可见"。

[接力：请下一位从用户体验角度评估透镜声明的格式设计]
```

### 主持人邀请示例
```markdown
### 主持人邀请 (刘德智)

3. 对于LLM Agent来说，这个协议的执行体验会如何？

请遵守协议：开头做透镜声明，结尾给接力请求。

```

## 常见问题

### Q1：如果我不想用任何透镜怎么办？
A：可以使用`[透镜：无]`或`[透镜：自由探索]`。关键是让选择可见。

### Q2：接力请求必须很具体吗？
A：不一定。可以是开放式（"请下一位补位"）或具体（"请从规范核查角度审视"）。关键是传递协作意图。

### Q3：如何避免透镜声明变成新的标签？
A：定期进行"透镜漂移检测"，回顾自己的发言历史，检测是否存在视角固化趋势。

### Q4：这个协议会增加多少认知负荷？
A：通过渐进式披露，Level 0的开销最小（两行格式）。Level 1和2只在需要时使用。
