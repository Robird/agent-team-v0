## 场势快照模板

### 四格完整版

```markdown
#### 场势快照

- **共识**：此刻我们已经同意什么？
- **分歧**：最大的分歧是什么？（没有就写"无"）
- **缺口**：还缺哪条关键输入/验证？需要谁补？
- **能量**：讨论节奏是过热/平稳/疲惫？
```

### 三格简化版

```markdown
#### 场势快照

- **共识**：...
- **缺口**：...
- **能量**：...
```

### 带透镜标注版

```markdown
#### 场势快照（本轮透镜：🔴 价值红）

- **共识**：我们都同意这个问题值得解决
- **分歧**：价值优先级（P0 vs P1）
- **缺口**：需要用户场景数据验证价值假设
- **能量**：讨论热烈，需要收敛 🌡️
```

### 失稳信号版

```markdown
#### 场势快照

- **共识**：...
- **分歧**：...
- **缺口**：...
- **能量**：过热 🔥

**失稳信号**：讨论进入人身推断，情绪升级
**托住动作**：时间盒5分钟冷却，回到事实层面
```

## 结论同步块模板

### 标准版

```markdown
#### 结论同步块

**✅ 已定**：
- 问题定义：...
- 核心原则：...
- 技术选型：...

**❓ 未定**：
- 具体实现细节：...
- 时间安排：...

**📋 行动项**（owner / 截止 / 验收）：
- [ ] @某人：完成XX文档（01-05前 / 可评审）
- [ ] @某人：调研YY方案（01-07前 / 报告）

**🔺 待裁决**（需要人类/外部输入）：
- 预算分配：需要监护人批准
- 优先级冲突：需要战略对齐
```

### 简化版

```markdown
#### 结论同步块

**✅ 已定**：
- ...

**❓ 未定**：
- ...

**📋 行动项**：
- ...
```

### 无行动项版

```markdown
#### 结论同步块

**✅ 已定**：
- 问题已澄清，无需进一步行动
- 结论记录供未来参考

**❓ 未定**：无
**📋 行动项**：无
**状态**：仅记录，无后续行动
```

## 邀请发言模板

### 标准邀请格式

```yaml
# @<AgentName>
taskTag: "#review" # 或 #design, #decision, #jam
chatroomFile: "agent-team/meeting/YYYY-MM-DD-<topic>.md"
appendHeading: "### <AgentName> 发言"
scope: "<ScopeDescription>"
outputForm: "<OutputFormat>"
```

### 带目标文件

```yaml
# @<AgentName>
taskTag: "#review"
chatroomFile: "agent-team/meeting/YYYY-MM-DD-<topic>.md"
targetFiles:
  - "<FilePath>"
appendHeading: "### <AgentName> 发言"
scope: "<ScopeDescription>"
outputForm: "<OutputFormat>"
```

### 实施邀请

```yaml
# @<AgentName>
taskTag: "#design"
chatroomFile: "agent-team/meeting/YYYY-MM-DD-<topic>.md"
appendHeading: "### <AgentName> 实施报告"
scope: "<ScopeDescription>"
implementationTarget: "<FilePath>"
outputForm: "实施报告模板"
```

### 开放式邀请

```yaml
# @<AgentName>
taskTag: "#jam"
chatroomFile: "agent-team/meeting/YYYY-MM-DD-<topic>.md"
appendHeading: "### <AgentName> 发言"
scope: "<ScopeDescription>"
outputForm: "任何形式"
```

## 实施报告模板

### 完整版

```markdown
### <Name> 实施报告

**实施任务**：<任务描述>
**目标文件**：`<path>`
**完成状态**：✅ 已完成 / ⚠️ 部分完成 / ❌ 失败

**实施内容**：
1. 修改了XX文件，添加了YY功能
2. 创建了ZZ文档，记录了设计决策
3. 更新了测试用例，覆盖新场景

**具体变更**：
```diff
// 示例代码变更
- const oldFunction = () => {};
+ const newFunction = (param: Type) => {
+   // 新实现
+ };
```

**待完善点**：
- 需要补充错误处理
- 文档需要进一步细化
- 测试覆盖率可提升

**下一步建议**：
- 建议进行代码评审
- 建议更新相关文档
- 建议添加集成测试

**验证方式**：
- [ ] 编译通过
- [ ] 测试通过
- [ ] 文档更新
- [ ] 代码评审完成
```

### 简化版

```markdown
### <Name> 实施报告

**任务**：<简短描述>
**文件**：`<path>`
**状态**：✅ 完成

**变更摘要**：
- 添加了XX功能
- 修复了YY问题
- 更新了ZZ文档

**待办**：
- ...

**验证**：编译/测试通过
```

## 使用指南

### 模板选择原则

| 场景 | 推荐模板 | 理由 |
|:-----|:---------|:-----|
| **正式评审** | 完整版会议记录 + 标准结论块 | 记录完整，便于审计 |
| **快速讨论** | 简化版会议记录 + 简化结论块 | 效率优先，减少开销 |
| **实施跟进** | 完整实施报告 | 变更可追溯，验证明确 |
| **学习练习** | 所有模板试用 | 熟悉各种格式 |

### 模板定制建议

1. **不要机械复制**：根据具体情境调整模板
2. **保持一致性**：团队内使用相同模板变体
3. **渐进式采用**：新手从简化版开始，熟练后用完整版
4. **反馈迭代**：根据使用反馈优化模板

### 模板维护

- 模板版本与主持指南版本同步
- 重大变更需团队评审
- 使用问题记录到 issue 系统
