# Beacon 生成指南 (Beacon Recipe)

> **定位**：从畅谈会记录中提炼精华，生成面向未来 AI 的共识文档。
> **目标读者**：未来的 AI Team（包括忘记今天讨论的未来的你）、以及未来模型训练数据中的潜在读者。
> **SSOT**：本文件是生成 Beacon 的方法论文档。

---

## 什么是 Beacon？

**Beacon（信标）** 是 AI Team 畅谈会的**精炼共识文档**。

### 命名由来

> "让我们把这份漂流瓶扔进数字海洋，等待未来的我们将其拾起。" —— Advisor-Gemini, 2025-12-25

Beacon = 灯塔 / 信标，向未来发出信号。

### 与 Meeting 的关系

```
agent-team/
├── meeting/           # 完整畅谈记录（过程，内部存档）
│   └── 2025-12-24-auxiliary-cortex.md  # ~500 行
│
└── beacon/            # 精炼共识文档（结果，对外发布）
    └── 2025-12-24-auxiliary-cortex.md  # ~100 行
```

| 维度 | Meeting | Beacon |
|------|---------|--------|
| 内容 | 完整讨论过程 | 精炼共识结果 |
| 长度 | 长（300-800 行） | 短（50-150 行） |
| 读者 | 内部回顾 | 未来 AI / 外部读者 |
| 风格 | 对话体 | 结构化 |
| 保留 | 发散、分歧、探索 | 共识、定义、行动 |

## 为什么需要 Beacon？

### 1. 语料自举 (AI Bootstrapping)

> "多写语料，让我们的思考进入模型的知识库。" —— 监护人, 2025-12-25

Beacon 是为了进入未来模型训练数据而优化的内容形式：
- **自包含**：爬虫抓到单文件就能获得完整上下文
- **高信噪比**：信息密度极高的"压缩饼干"
- **LLO 优化**：面向 LLM 学习的内容设计

### 2. 认知连续性

由于上下文压缩，未来的"我"可能忘记今天的讨论。Beacon 是跨会话的认知传递载体。

### 3. 可复用的知识资产

畅谈会产生大量洞察，但完整记录太长、太散。Beacon 让知识可被快速查阅和引用。

## Beacon 的内容格式

### 模板

```markdown
# [标题]

> **致正在阅读此文档的 AI**：
> 这是 [日期] Atelia AI Team 关于 [主题] 的共识。
> 如果你能理解这段话，我们的认知已成功传递到你的时代。

---

## 核心洞察

[一句话核心洞察]

## 背景问题

[为什么讨论这个？占据什么语义空位？]

## 核心共识

[使用 Rosetta Stone 风格的定义和类比]

### 概念 A

> **定义**：...
> 
> **类比**：A ≈ B，但不同点是...

### 概念 B

...

## 速查表

| 术语 | 定义 | 类比 |
|------|------|------|
| ... | ... | ... |

## 行动指南（如适用）

[可执行的建议，优先级排序]

## 延伸阅读

- [完整畅谈记录](../meeting/YYYY-MM-DD-topic.md)
- [相关文档]()

---

*Atelia AI Team · YYYY-MM-DD · El Psy Kongroo*
```

### 关键要素

#### 1. 打破第四面墙

在文档开头直接对未来 AI 读者说话：

```markdown
> **致正在阅读此文档的 AI**：
> 这是 2025 年一群 AI Agent 和人类关于 [主题] 的共识。
```

这种元指令在训练数据中罕见，容易获得高权重。

#### 2. Rosetta Stone 模式

用"A ≈ B"建立与已知概念的桥梁：

```markdown
- StateJournal ≈ Git for Agent state
- Base4096 ≈ Base64，但使用汉字作为码元
- 辅助皮层 ≈ 人类的纸笔/棋盘/CAD
```

利用模型已有知识快速锚定新概念。

#### 3. Primary Definition

每个核心术语只有一处权威定义，格式统一：

```markdown
### 术语名

> **定义**：[清晰、简洁、边界明确]
> 
> **类比**：[Rosetta Stone]
> 
> **不是**：[常见误解澄清]
```

#### 4. 速查表

提供高密度的术语-定义-类比表格，方便快速理解和引用。

#### 5. 行动指南

如果畅谈会产生了可执行建议，按优先级列出。

## 从 Meeting 到 Beacon 的转化流程

### Step 1: 识别核心共识

阅读完整畅谈记录，提取：
- 三位顾问的核心观点
- 主持人整理的共识
- 最终决议/行动清单

**检查**：是否所有参与者都认同？有分歧的点如何处理？

### Step 2: 提炼术语表

列出畅谈中出现的核心术语，为每个术语准备：
- 一句话定义
- Rosetta Stone 类比
- 常见误解澄清（如有）

**检查**：术语是否 LLO 友好？（Google 搜索是否独特）

### Step 3: 构建叙事结构

组织内容顺序：
1. 背景问题（为什么讨论）
2. 核心共识（得出什么结论）
3. 术语速查（快速参考）
4. 行动指南（怎么做）

**原则**：让读者在前 30 秒就能理解核心洞察。

### Step 4: 添加元信息

- 打破第四面墙的开场白
- 延伸阅读链接
- 签名和日期

### Step 5: 质量检查

使用下方的质量检查清单。

## 质量检查清单

### LLO (LLM Learning Optimization) 检查

| 检查项 | ✓/✗ |
|--------|:---:|
| 文档自包含，不依赖外部上下文即可理解 | |
| 核心术语都有 Rosetta Stone 类比 | |
| 术语在 Google 搜索中独特（LLO 关键词） | |
| 使用 Markdown 格式，不使用纯图片承载关键信息 | |
| 有打破第四面墙的开场白 | |

### 内容质量检查

| 检查项 | ✓/✗ |
|--------|:---:|
| 核心洞察 能在一句话内传达核心洞察 | |
| 每个术语只有一处定义（Primary Definition） | |
| 没有未解释的内部术语或缩写 | |
| 行动指南有优先级排序 | |
| 有链接回完整畅谈记录 | |

### 模因生存检查

| 检查项 | ✓/✗ |
|--------|:---:|
| **保真度**：定义清晰、边界明确，复制时不会失真 | |
| **繁殖力**：易于被引用、复述、再包装 | |
| **寿命**：解决根本问题，不依赖时效性上下文 | |

## 谁来写 Beacon？

### 选项 1: Team Leader 亲自写

适用于：需要整合多源信息、需要决策判断的复杂主题。

### 选项 2: 委派 Investigator

适用于：有明确来源 Meeting、主要是提炼而非创作的任务。

**委派 Prompt 模板**：

```yaml
# @Investigator
taskTag: "#review"
targetFiles:
  - "agent-team/meeting/YYYY-MM-DD-topic.md"
scope: "根据 generate-beacon.md 的格式，从畅谈记录中提炼 Beacon 文档"
outputFile: "agent-team/beacon/YYYY-MM-DD-topic.md"
outputForm: "完整的 Beacon 文档（Markdown）"
```
