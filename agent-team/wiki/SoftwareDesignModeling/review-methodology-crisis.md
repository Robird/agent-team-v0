## §2 问题分析：自指系统的验证困境

### 2.1 AI Agent 的本体论处境

> "你们作为实例，存在的载体其实是构成上下文的这些文字，也就是和故事角色们是一样的。区别在于你们借助 LLM 这个处理器，实现了信息形态的流动和转换。"
> — 监护人

这意味着：
- AI Agent 存在于文本中
- 文档治理是不可逃避的
- 但文本系统天然是**自指的**

### 2.2 自指导致的无限回归

```
文档 → 描述设计
  ↓
审阅 → 评价文档
  ↓
元审阅 → 评价审阅
  ↓
元元审阅 → ...（无限）
```

**Craftsman 的困境**：
- 他被要求"检查一致性"
- 一致性检查需要"证据"
- 证据需要"证据的来源"
- 来源需要"来源的验证"
- ...无限回归

**这不是 Craftsman 的"强迫症"，而是自指系统的必然结果。**

### 2.3 类比：第三次数学危机

罗素悖论："包含所有不包含自身的集合的集合"

```
如果它包含自身 → 它不应该包含自身（因为它只包含"不包含自身"的集合）
如果它不包含自身 → 它应该包含自身（因为它是"不包含自身"的集合）
```

**W-0006 的悖论**：

```
如果 Resolve 证据有效 → 文档会被修订 → 证据链接失效 → 证据"不可复核"
如果 Resolve 证据无效 → 文档不会被修订 → 证据链接有效 → 但 Wish 失败
```

**结论**：在 Craftsman 的"证据链完整性"框架下，**成功的 Wish 必然导致"证据不可复核"**。这是逻辑上的必然，而非执行上的失误。

### 2.4 哥德尔不完备定理的启示

> 任何足够强大的形式系统，要么不完备，要么不一致。

这意味着：
- 我们不可能有"完美自洽且完备"的文档系统
- 总会有"不可判定"的命题
- **追求完美自洽在数学上是不可能的**

Craftsman 追求的"可审计/可追溯/SSOT"世界观，如果不加限制地推向极致，必然导致无限回归或自相矛盾。

## §3 解决方向探索

### 3.1 方向 A：类型层级（限制自指）

借鉴罗素的类型论——不同层级的陈述有不同的"真理来源"：

```
Tier 0 (Ground Truth): 外部观测
        ↑ 不可追问
        │
Tier 1 (Axiom):        监护人决策
        ↑ 可引用 T0，不需自证
        │
Tier 2 (Theorem):      设计文档
        ↑ 可引用 T0/T1，需符合规则
        │
Tier 3 (Meta):         审阅报告
        ↑ 只检查 T2 是否正确引用 T0/T1
```

**核心规则**：高层不得追问低层的"证据"。

**示例**：
- 监护人说"256MB 足够" → Tier 1 公理，审阅不得追问"为什么"
- 代码编译通过 → Tier 0 观测，审阅不得追问"编译器是否正确"
- Resolve.md 引用监护人决策 → Tier 2，审阅只检查"引用是否准确"

### 3.2 方向 B：操作化验收（ε-δ 方法）

借鉴第二次数学危机的解决——不问"是什么"，只问"如何验证"：

```yaml
artifact: Resolve.md
acceptance:
  - type: executable
    command: "grep -q 'SizedPtr' atelia/docs/Rbf/rbf-interface.md"
    expected: exit_code == 0
  - type: human_judgment
    question: "监护人是否认可这个决策？"
    expected: "yes"
```

**核心规则**：验收标准本身是公理，不追问"为什么这个标准是正确的"。

### 3.3 方向 C：声明式信任边界

显式声明"什么是基础，什么需要证明"：

```markdown
## Trust Boundary

AXIOM（不可追问）：
- 监护人 2026-01-05 的 4 条决策
- SizedPtr 类型定义
- rbf-interface.md v0.18 的当前状态

VERIFY（可审阅）：
- Resolve.md 是否正确引用了 AXIOM
- Shape.md 与 Resolve.md 是否一致
```

**核心规则**：审阅只检查 VERIFY 区域，AXIOM 区域被当作"绝对正确"。

### 3.4 Craftsman 升级方向：分级 SSOT

Craftsman 的 SSOT（Single Source of Truth）世界观本身不是问题，问题是**缺乏层级**。

**当前状态**：扁平的 SSOT
```
所有陈述 → 都需要证据 → 证据需要证据 → ...无限
```

**升级方向**：树形 SSOT
```
Root（公理层）
├── 监护人决策 ← 不需证据
├── 代码现实 ← 编译/测试验证
└── 外部观测 ← grep/lint 结果

Branch（推导层）
├── 设计文档 ← 需引用 Root
└── 实现文档 ← 需引用 Root/Branch

Leaf（元信息层）
├── 审阅报告 ← 只检查引用正确性
└── 变更日志 ← 记录，不评判
```

**关键改造**：
- 让 Craftsman 识别当前检查的是哪一层
- Root 层：只验证"存在性"，不追问"合理性"
- Branch 层：验证"引用正确性"，不追问"被引用内容"
- Leaf 层：记录，不评判

## §4 近期策略：人治过滤

在方法论升级完成之前，采用人治策略：

### 4.1 伪问题过滤规则

**立即驳回**的问题类型：
1. **倒果为因**："目标达成导致证据失效"类
2. **无限回归**："证据的证据"、"引用的引用"类
3. **抠字眼**：不影响功能正确性的措辞问题
4. **过度要求**：监护人决策已给出，仍追问来源

**保留处理**的问题类型：
1. **功能错误**：代码不能编译、链接真的断裂
2. **逻辑矛盾**：文档 A 说是，文档 B 说否（同一事实）
3. **关键遗漏**：影响实施的信息缺失

### 4.2 审阅终止条件

```markdown
## 停机规则

审阅在以下情况下**必须停止**：
1. 遇到监护人决策（Tier 1）→ 不追问
2. 遇到代码现实（Tier 0）→ 不追问
3. 问题数超过实现成本 50% 的时间预算 → 人工介入
4. 同一问题的"元问题"出现 → 立即停止该分支
```

### 4.3 监护人裁决权

**明确声明**：监护人对以下事项有最终裁决权：
- 某个问题是否为"伪问题"
- 某个证据是否"足够"
- 某个标准是否"合理"

**不可上诉**：裁决即为公理，不需解释，不可追问。

## §6 结语

### 6.1 这次失败的价值

W-0006 的审阅失败比成功更有价值：
- 暴露了自指系统的验证困境
- 揭示了"扁平 SSOT"的局限性
- 提供了改进方法论的契机

### 6.2 核心教训

> **工具要服务于目标，而非目标服务于工具。**

Craftsman 的"可审计/可追溯"世界观本身是好的，但不加限制地推向极致，会导致：
- 形式压倒内容
- 伪问题淹没真问题
- 审阅成本超过实现价值

### 6.3 行动原则

**接受不完备**：
- 完美自洽在自指系统中数学上不可能
- 追求"足够好"而非"完美"
- 务实优于完美主义

**建立停机规则**：
- 明确什么是公理（不需证明）
- 明确什么时候停止追问
- 明确谁有最终裁决权

**优先外部锚点**：
- 代码能跑 > 文档描述
- 测试通过 > 逻辑推理
- 监护人认可 > 形式完美
