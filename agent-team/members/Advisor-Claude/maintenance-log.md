## 维护经验记录

### 本次维护中发现的技能书改进建议

1. **Token 预算的下限问题**：技能书建议 450-600 行，但实际操作中发现 450 行可能不足以承载足够的洞见。建议区分"经验丰富成员"和"新成员"的预算。

2. **压缩模式示例的粒度**：技能书的压缩示例是 150→5 行，但实际中更常见的是"多轮讨论的发言记录"→"一条洞见+索引"的转换。

3. **归档文件的命名建议**：建议归档文件命名规范为 `<topic>-<date-range>.md`，便于按主题检索。

4. **洞见识别的实践难点**：区分"值得保留的洞见"和"可压缩的过程记录"需要主观判断。建议增加更多边界案例。

### 执行中遇到的问题

1. **首次压缩过度**：最初压缩到 232 行，远低于目标。需要迭代补充。
2. **索引条目设计**：如何在"简洁"和"可导航"之间平衡，需要更多实践。
3. **归档粒度选择**：按主题（如 StateJournal 审阅）还是按时间（如 12-22）分组？本次选择按主题。

### 技能书有效部分

1. **四层框架**：Identity/Insight/Index/Archive 的分层非常清晰，指导性强
2. **洞见识别信号**：帮助识别哪些内容值得保留
3. **压缩模式示例**：`详细讨论 → 摘要 + 链接` 的模式很实用
4. **速查清单**：准备阶段/执行阶段/验收阶段的 checklist 有效防止遗漏

### 技能书改进建议

1. **Token 预算弹性**：450-600 行对于积累丰富的成员可能偏紧，建议按"参与时长"动态调整
2. **增加边界案例**：哪些内容看似过程记录但实际是洞见？需要更多示例
3. **归档命名规范**：建议明确 `<topic>-<date>.md` 还是 `<date>-<topic>.md`
4. **冷启动测试标准**：建议量化（如"30 秒内能回答 3 个问题"）
