# Wish: 实验-响应驱动畅谈会

> **一句话动机**: 当前的畅谈会形式无法处理“邀请大家根据当前各自的记忆文档，为`agent-team/how-to/maintain-external-memory.md`提建议”这样的情形，主持人只会问"这本质上是什么？"/"这会如何被体验？"/"这在实践中如何落地？"这类跑题的僵化邀请词。我们需要新的**需求与贡献**驱动的畅谈会。

## 目标与边界

**目标 (Goals)**:
- [ ] 挖掘与设计出一组建设性的畅谈会**贡献动作**，也就是我们项目/计划的**能力需求**，例如：理解需求、分析、调查、指出盲点与误区、总结、对比、审阅（合理性、一致性、自洽性）、发掘问题、提供知识、修订文档、设计算法、设计方案、编写源码、扮演用户体验并反馈、运行测试。
- [ ] 若畅谈会上发现一时难以解决的问题，则记录到相应的Artifact-Tier的文档中。
- [ ] 若畅谈会上形成一致共识，则沉淀为相应Artifact-Tier的产物，例如文档和代码。
- [ ] 让畅谈会基于**能力需求**与**贡献动作**之间的震荡来驱动，迭代进行挖掘能力需求->选择最能够执行贡献动作的人。
- [ ] 形成由各AI Team成员构成的**能力池**，形如一个markdown list或表格，记录了每个成员所能提供的优势能力，鼓励能力重叠。能力主要来自于4个方面：模型知识库与技能训练；系统提示词能力激活和预制行为模式；经历积累(例如对项目的了解、教训与成功经验)；掌握的工具
- [ ] 畅谈会主持人的主要职能变为{归纳与整理当前已知的**能力需求**}，根据**能力池**选择最适合的贡献提供者。在形成共识或发现僵局之前，开放性的邀请能力提供者回应前人的发言(典型情况下提示词就是“阅读畅谈会文件，你对前面的发言有哪些想法或思路？请将发言追加到文件末尾”)。在形成共识或发现僵局之后，明确要求提供能力，如修订文档，编写代码，调查情况，提供知识等。
- [ ] 保留当前`agent-team/how-to/run-jam-session.md`中识别层级和层级错位的能力“当前在哪个Tier？是否存在错位（如A谈Craft，B谈Resolve）？”。
- [ ] 保留当前畅谈会的动态性。
- [ ] 修订`agent-team/how-to/run-jam-session.md`，落实成可操作性强，文本精简干练的文档。复杂的分析和构建脚手架应独立到另外的文档中，构造过程可以复杂，但成品Recipe需要形式上简单。

**非目标 (Non-Goals)**:
- 尚未明确

## 验收标准 (Acceptance Criteria)

- [ ] 尚未完全明确，但初步实验思路是组织一系列新的畅谈会并观察过程，目标是以推进`atelia/docs/Data/Draft/SizedPtr.md`在`atelia/src/Dataatelia/src/Data`中实现，并在`atelia/docs/Rbf/rbf-interface.md`文档中替代`Address64`类型。
